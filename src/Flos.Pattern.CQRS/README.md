# Flos.Pattern.CQRS

CQRS + Event Sourcing pattern for discrete, auditable state transitions. Ideal for turn-based, card, strategy, and board games.

## Installation

```xml
<PackageReference Include="Flos.Pattern.CQRS" />
```

## Quick Start

A complete example showing the command → handler → applier → pipeline flow:

```csharp
using Flos.Core.Errors;
using Flos.Core.Module;
using Flos.Core.Sessions;
using Flos.Core.Scheduling;
using Flos.Core.State;
using Flos.Pattern.CQRS;

// 1. Define state
[DeepClone]
public class ScoreState : IStateSlice, IDeepCloneable<ScoreState>
{
    public int Points { get; set; }
    // DeepClone() auto-generated by Flos.Generators
}

// 2. Define a command and event (readonly record struct for zero-allocation)
public readonly record struct AddPointsCommand(int Amount) : ICommand;
public readonly record struct PointsAddedEvent(int Amount) : IEvent;

// 3. Handler: validates against read-only state, emits events
public class AddPointsHandler : ICommandHandler<AddPointsCommand>
{
    public ErrorCode Handle(AddPointsCommand cmd, IStateReader state, EventBuffer events)
    {
        if (cmd.Amount <= 0)
            return GameErrors.InvalidAmount;

        events.Add(new PointsAddedEvent(cmd.Amount));
        return default; // success
    }
}

// 4. Applier: mutates state in response to events
public class PointsAddedApplier : IEventApplier<PointsAddedEvent, ScoreState>
{
    public void Apply(ScoreState state, PointsAddedEvent evt)
    {
        state.Points += evt.Amount;
    }
}

// 5. Module: wires everything together
public class ScoreModule : ModuleBase
{
    public override string Id => "Score";
    public override IReadOnlyList<string> RequiredPatterns => [CQRSPattern.Id];

    public override void OnLoad(IServiceRegistry scope)
    {
        base.OnLoad(scope);
        Scope.Resolve<IWorld>().Register(new ScoreState());
    }

    public override void OnInitialize()
    {
        // Register slices for snapshot support
        Scope.Resolve<ISnapshots>().RegisterSlice<ScoreState>();

        var registry = Scope.Resolve<IHandlerRegistry>();
        registry.Register(new AddPointsHandler());
        registry.Register<PointsAddedEvent, ScoreState>(new PointsAddedApplier());
    }
}

// 6. Run
var session = new Session();
session.Initialize(new SessionConfig
{
    Modules = [
        new SnapshotModule(),
        new CQRSPatternModule(),
        new ScoreModule(),
    ],
    TickMode = TickMode.StepBased,
});
session.Start();

var pipeline = session.RootScope.Resolve<IPipeline>();
var result = pipeline.Send(new AddPointsCommand(10));
// result.IsSuccess == true, ScoreState.Points == 10
```

## Quick Usage

```csharp
// Define a command and event
public readonly record struct DrawCardCommand(int Count) : ICommand;

public readonly record struct CardDrawnEvent(string CardId) : IEvent;

// Implement a handler (validates against read-only state, emits events via EventBuffer)
public class DrawCardHandler : ICommandHandler<DrawCardCommand>
{
    public ErrorCode Handle(DrawCardCommand cmd, IStateReader state, EventBuffer events)
    {
        var deck = state.Get<DeckState>();
        if (deck.Cards.Count < cmd.Count)
            return GameErrors.InsufficientCards;

        for (int i = 0; i < cmd.Count; i++)
            events.Add(new CardDrawnEvent(deck.Cards[i]));
        return default; // success
    }
}

// Implement an applier (mutates state)
public class CardDrawnApplier : IEventApplier<CardDrawnEvent, DeckState>
{
    public void Apply(DeckState state, CardDrawnEvent evt) => state.Cards.RemoveAt(0);
}
```

## Module Setup

`CQRSPatternModule` is the entry point. Add it to your session's module list:

```csharp
Modules = [
    new SnapshotModule(),       // provides ISnapshots for rollback
    new CQRSPatternModule(),    // registers IPipeline, IHandlerRegistry, IEventJournal
    new MyGameModule(),         // your game logic
]
```

`CQRSPatternModule` registers the `CQRSPattern.Id` pattern, the default `IPipeline` and `IHandlerRegistry`, the `IEventJournal`, and installs the `DeferredCQRSMiddleware` on the message bus.

To customize pipeline configuration, register a `CQRSConfig` instance before `CQRSPatternModule` runs:

```csharp
scope.Register(new CQRSConfig
{
    FaultMode = ApplierFaultMode.Tolerant,
    EnableJournal = false,
    EnableRollback = true,
    MaxJournalEntries = 1000,  // 0 = unbounded (default)
    MaxDeferralDepth = 16,     // 0 = disable deferred sends (default 16)
});
```

## API Overview

| Type | Description |
|------|-------------|
| `ICommand` | Command marker interface (extends `IMessage`) |
| `IEvent` | Domain event marker |
| `ICommandHandler<T>` | Validates commands against read-only state, emits events via `EventBuffer` |
| `IEventApplier<TEvent, TState>` | Mutates state in response to events |
| `IPipeline` | Sends commands: `Send(ICommand)` (boxed) and `Send<TCommand>(TCommand)` (zero-alloc). Returns `Result<EventBuffer>`. |
| `IHandlerRegistry` | Registers handlers (`Register<TCommand>`) and appliers (`Register<TEvent, TState>`) |
| `IEventJournal` | Append-only event log indexed by tick |
| `IRollbackProvider` | State capture and rollback for the pipeline. `Capture`, `RestoreTo`, `Release` |
| `ICQRSAdapter` | Adapter interface for custom pipeline implementations. `CreatePipeline(IMessageBus, IWorld)` returns `(IPipeline, IHandlerRegistry)`. |
| `CQRSPatternModule` | Module entry point. Registers pattern, pipeline, journal, and middleware. |
| `CQRSConfig` | Runtime config: `FaultMode`, `EnableJournal`, `EnableRollback`, `MaxJournalEntries`, `MaxDeferralDepth` |
| `ApplierFaultMode` | `Strict` (default), `Tolerant`, `Fatal` |
| `EventBuffer` | Reusable, array-backed buffer for events emitted by handlers. Owned by the pipeline. |
| `CommandRejectedMessage` | Published on the bus when a command fails validation or application. Carries `ICommand` and `ErrorCode`. |
| `DeferredCQRSMiddleware` | (internal) Middleware that intercepts `ICommand` messages published via `bus.Publish()` and routes them through the pipeline. |

## Pipeline Execution

1. Route command to handler (fail: return `CQRSErrors.UnknownCommand`)
2. Capture read-only snapshot via `IRollbackProvider` (if `EnableRollback` is true)
3. Handler validates against `IStateReader` and emits events into `EventBuffer` (fail: publish `CommandRejectedMessage`, return error)
4. Apply events to world state via registered appliers
5. Publish events on MessageBus
6. Journal events (if enabled)

## EventBuffer

`EventBuffer` is owned by the pipeline and reused between commands. It is passed to handlers via `Handle(cmd, state, events)`. Handlers append events with `events.Add(new MyEvent(...))`.

The `EventBuffer` returned by `pipeline.Send()` is valid only until the next `Send()` call — the pipeline resets the buffer between commands. If you need to retain event data, copy it before the next send.

## Bus-Published Commands

Commands published via `bus.Publish(new MyCommand())` are automatically intercepted by `DeferredCQRSMiddleware` and routed through the pipeline. This is convenient for fire-and-forget scenarios but has two caveats:
- The command is boxed (goes through the non-generic `IPipeline.Send(ICommand)` path)
- The `Result<EventBuffer>` is not returned to the caller — failures are logged as warnings

For zero-allocation dispatch and result access, call `pipeline.Send<TCommand>(cmd)` directly.

## Error Codes

| Code | Description |
|------|-------------|
| `CQRSErrors.UnknownCommand` | No handler registered for the command type |
| `CQRSErrors.HandlerFailed` | The handler returned a non-default `ErrorCode` |
| `CQRSErrors.ApplierFailed` | An event applier threw an exception during `Apply` |
| `CQRSErrors.ReentrantSend` | `Pipeline.Send` called reentrantly with `MaxDeferralDepth=0` (deferred sends disabled) |
| `CQRSErrors.DeferralDepthExceeded` | Deferred command queue exceeded `MaxDeferralDepth`. Possible infinite command loop |
| `CQRSErrors.InvalidConfiguration` | `EnableRollback` is true but no `IRollbackProvider` is registered |

## ApplierFaultMode

| Mode | Behavior |
|------|----------|
| `Strict` (default) | Rolls back world state to pre-command snapshot and returns a failure result. |
| `Tolerant` | Skips the failing applier and continues. State changes from successful appliers are kept. Increments a fault counter. Command still succeeds. |
| `Fatal` | Throws a `FlosException` immediately, crashing the session. |

## Custom Pipeline

To replace the built-in pipeline, implement `ICQRSAdapter` and register it before `CQRSPatternModule` loads:

```csharp
public class MyAdapter : ICQRSAdapter
{
    public (IPipeline Pipeline, IHandlerRegistry Registry) CreatePipeline(IMessageBus bus, IWorld world)
    {
        var pipeline = new MyCustomPipeline(bus, world);
        return (pipeline, pipeline); // both may be the same object
    }
}

// In your module's OnLoad (must run before CQRSPatternModule):
scope.Register<ICQRSAdapter>(new MyAdapter());
```

`CQRSPatternModule` uses `TryRegister` for `ICQRSAdapter`, so a pre-registered adapter takes priority over the built-in one.

## Deferred Send (Reentrant Commands)

When `Send` is called from within a handler, applier, or event subscriber, the command is **queued** and executed in FIFO order after the outermost `Send` completes. This enables chain-reaction patterns (e.g., Battlecry triggers Deathrattle triggers end-of-turn effects) without breaking pipeline invariants.

```
pipeline.Send(PlayCardCommand)           // outermost Send
  -> Handler emits CardPlayedEvent
  -> Applier: card enters play
  -> Event published -> subscriber calls pipeline.Send(BattlecryCommand)  // queued
  -> outermost Send completes
  -> drain: Send(BattlecryCommand)
    -> Handler emits DamageDealtEvent
    -> Applier: target takes damage, dies
    -> Event published -> subscriber calls pipeline.Send(DeathrattleCommand)  // queued
    -> BattlecryCommand completes
  -> drain: Send(DeathrattleCommand)
    -> ...
```

Key properties:
- **Breadth-first execution:** deferred commands execute in FIFO order, not recursively
- **Independent rollback:** each deferred command has its own rollback snapshot
- **Fresh state view:** each deferred handler sees the latest world state (including mutations from prior commands)
- **Depth limit:** `MaxDeferralDepth` (default 16) prevents infinite loops; throws `CQRSErrors.DeferralDepthExceeded` when exceeded
- **Opt-out:** set `MaxDeferralDepth = 0` to disable deferred sends and throw `CQRSErrors.ReentrantSend` immediately (original behavior)
- **Return value:** the deferred caller receives `Result<EventBuffer>.Ok` with the outer command's event buffer; callers inside handlers should not rely on the returned events

## Determinism Notes

- Handlers receive `IStateReader` (read-only snapshot when rollback is enabled) — cannot accidentally mutate state.
- Applier exceptions trigger automatic state rollback in `Strict` mode (configurable via `ApplierFaultMode`).
- Event journal enables deterministic replay.
- Deferred commands are journaled in execution order, preserving deterministic replay across chain reactions.
